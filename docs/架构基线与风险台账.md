# 架构基线与风险台账

本文档不再承担按周排期的“路线图”职责，而是用于长期沉淀三类信息：

- 当前项目的架构基线
- 仍然成立的核心风险
- 后续优化时建议优先处理的顺序

如果项目结构、部署方式或主要技术债已经发生变化，应优先更新本文件。

## 1. 当前架构基线

### 1.1 技术与运行拓扑

- 前端：Vue 3 + Vite + TypeScript + Pinia + Element Plus
- 后端：.NET 9 + ASP.NET Core + EF Core + MySQL
- 缓存/队列：嵌入式 Garnet（Redis 协议）
- 本地开发：
  - 前端：`http://localhost:5200`
  - 后端：`http://localhost:3001`
- 容器部署：宿主机 `5200 -> 容器 3001`

### 1.2 目录边界

- 前端主工作区：`src/`
- 后端主工作区：`server_net/`
- 运行时数据目录：`server/data/`
- 录制归档：
  - 新场次：`server/data/danmaku/{uid}/*.jsonl`
  - 历史场次：`server/data/danmaku/` 下既有的 `XML` 文件
- 部署与说明文档：`docs/`

### 1.3 前端关键模块

- `src/main.ts`
  - 应用装配入口，挂载 `Pinia`、`router`、`Element Plus`
- `src/router/index.ts`
  - 路由主干，当前使用 Hash 路由
- `src/api/danmaku.ts`
  - API 访问层与字段兼容层
- `src/stores/danmakuStore.ts`
  - 前台核心状态管理
- `src/views/AdminView.vue`
  - 管理后台聚合页面，当前仍是大单文件

### 1.4 后端关键模块

- `server_net/Program.cs`
  - 启动装配、服务注册、静态资源映射、控制器映射
- `server_net/Controllers/AdminController.cs`
  - 后台管理接口
- `server_net/Controllers/DanmakuController.cs`
  - 前台弹幕、主播、场次相关接口
- `server_net/Services/`
  - 录制、弹幕处理、调度、图片抓取、缓存等服务
- `server_net/Data/`
  - EF Core 上下文与初始化逻辑

### 1.5 部署基线

- 前端静态资源主要部署到腾讯云 COS + CDN
- 后端 API 主要部署在腾讯云服务器，通过 1Panel + Docker 提供服务
- 推荐域名：
  - `ovodm.top`
  - `www.ovodm.top`
  - `api.ovodm.top`

详细接入与上线操作见：

- `docs/域名接入与部署指南.md`

## 2. 关键数据流

### 2.1 前台主链路

1. 用户选择主播或场次
2. `danmakuStore.loadSession` 先拉取场次摘要
3. 页面优先渲染统计摘要
4. 用户继续查看明细时，`fetchDanmaku` 拉取首批弹幕
5. 直播中活跃场次优先从 Redis 事件队列读取，新归档场次从 `JSONL` 读取，历史场次回退 `XML`
6. 滚动到底部时，`loadMore` 继续分页追加
7. 搜索和筛选在前端 `filteredDanmaku` 中执行

### 2.2 管理后台链路

1. 用户在 `AdminView` 输入 token
2. 前端根据同源或跨域情况生成认证参数
3. 页面通过 `adminApi` 访问 `/admin/rooms`、`/admin/sessions`、`/admin/song-requests`
4. 页面内完成增删改查、分页、筛选和批量操作

### 2.3 部署与资源链路

1. 本地开发通过 Vite 代理 `/api` 到后端
2. 后端提供 API 与主播资源目录映射
3. 前端构建产物上传到 COS，后端主要负责 API 与运行时静态资源

## 3. 风险台账

下表中的风险只保留当前仍然成立、且会实质影响维护和迭代效率的问题。

| ID | 类别 | 优先级 | 状态 | 风险描述 | 影响范围 | 建议动作 |
| --- | --- | --- | --- | --- | --- | --- |
| R1 | 可维护性 | P1 | Open | `src/views/AdminView.vue` 集中了承载鉴权、监控、场次、点歌、分页、批量操作和样式 | 后台任何功能改动都容易产生连锁回归 | 按资源域拆为 composables + 子组件，先拆鉴权与三类资源访问 |
| R2 | 契约 | P1 | Open | `src/api/danmaku.ts` 仍存在大量 snake/camel/Pascal 字段兼容 | 接口变化难以及时暴露，前端持续背负适配债务 | 明确后端统一字段规范，逐步减少前端兼容分支 |
| R3 | 性能 | P2 | Open | `src/stores/danmakuStore.ts` 的 `loadMore` 仅有基础 loading 判断 | 高频滚动时可能出现重复请求和重复拼接 | 增加 in-flight page 去重和结果合并去重 |
| R4 | 性能 | P2 | Open | `filteredDanmaku` 对大列表做实时全量过滤 | 搜索输入和筛选切换会带来主线程压力 | 增加 debounce，必要时拆分实时和异步过滤路径 |
| R5 | 工程化 | P2 | Open | 当前缺少明确 lint/test/CI 护栏 | 回归发现偏后置，改动成本高 | 增加最小 lint、基础测试和 CI 校验 |
| R6 | 契约 | P2 | Open | 管理端认证仍同时存在 header/query 传递策略 | 不同网关或部署场景下可能产生语义分裂 | 收敛到单一权威传递方式，并补充部署说明 |
| R7 | 工程化 | P3 | Open | 文档职责曾有重叠，README、部署文档、架构文档边界容易漂移 | 接手成本上升，信息查找效率下降 | 持续维持 README 入口化、部署文档专题化、风险文档台账化 |

## 4. 推荐优化顺序

这里给出顺序，不给出固定周数。原因很简单：这类时间承诺很快过时，但优先级仍然有用。

### 优先级 1

- 建立基础工程护栏
  - 增加 `lint`
  - 增加最小单元测试
  - 让关键校验能进入 CI

### 优先级 2

- 拆分 `AdminView.vue`
  - 先抽 `useAdminAuth`
  - 再按 `rooms / sessions / songRequests` 拆资源域逻辑
  - 最后拆页面面板组件

### 优先级 3

- 优化弹幕列表性能
  - 补分页请求去重
  - 补结果去重
  - 加搜索 debounce

### 优先级 4

- 收敛前后端契约
  - 对齐字段命名
  - 对齐时间戳表达
  - 逐步减少前端兼容分支

## 5. 何时更新本文件

出现以下情况之一时，应更新本文件：

- `AdminView.vue` 已被完成拆分
- 前后端字段命名已统一
- 已增加 lint、test、CI 护栏
- 弹幕分页与过滤策略已完成优化
- 部署方式从当前 COS/CDN + 1Panel/Docker 方案迁移
- README、部署文档、协作文档职责再次变化
