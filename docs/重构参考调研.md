# 重构参考调研：弹幕录制与多账号架构

> 目的：为 `danmu_vue` 后端重构提供外部参考，重点关注多账号、录制生命周期、失败恢复、限流、以及 jsonl 归档策略。

## 1. 结论先行

### 1.1 是否需要修改现有 tmp → final 规则

**需要修改，但不需要推翻。**

建议把规则从“依赖数据库状态的归档”改成：

- **jsonl 是唯一的历史事实源**
- `tmp` 只作为**录制中 / 崩溃恢复**的临时缓冲
- `final` 是**可查询、不可变、稳定命名**的历史归档
- 归档动作必须**幂等**，不能依赖 DB 才判断“是否完成”

### 1.2 推荐的归档语义

1. 录制进行中：先写 `tmp`
2. 正常结束：将 `tmp` 收敛为 `final`
3. 异常退出：保留 `tmp` 供重启恢复
4. 重启恢复：优先从 `tmp` 补尾并继续，避免重复写入
5. 查询层：只面向 `final`，`tmp` 不参与常规查询

这样可以避免：

- 录制结束后 DB 状态和文件状态不同步
- 重启后丢最后一段弹幕
- 旧归档规则把“是否完成”绑定到数据库

## 2. 值得参考的开源项目

### 2.1 `danmakus-client-main`

仓库特征：

- TypeScript + Bun + Tauri
- 面向 B 站弹幕、房间调度、账号管理
- 有明确的 **AuthManager / CookieManager / Room Coordinator** 结构

可借鉴点：

- **账号池抽象清晰**：cookie 来源、优先级、切换策略、健康状态分离
- **运行时协调器**：房间分配、连接状态、调度状态不是散落在业务代码里
- **generation / runtime version** 思路：防止旧异步任务回写过时状态
- **断连与恢复**：更重视“连接状态机”而不是只看单次连接

适合迁移到本项目的模式：

- 统一账号选择入口
- 账号健康状态模型
- 过时操作保护（generation）

不适合直接照搬：

- 它更偏“客户端协调器”，不是服务端录制后台
- 连接池/Runtime 协作模型不适合当前每房间独立 recorder 的结构

### 2.2 `bililive-go`

仓库特征：

- Go 编写的多平台直播录制工具
- 目录清晰：`live/`、`recorders/`、`servers/`、`listeners/`
- 有成熟的启动、关闭、事件驱动和录制管理思路

可借鉴点：

- **优雅关闭**：录制器停止、清理、关闭流程明确
- **事件驱动**：连接、消息、结束、异常都通过事件流转
- **平台级限流**：不是所有请求都用同一把锁/同一个节流器
- **录制器状态广播**：状态变化对外可见

适合迁移到本项目的模式：

- 录制生命周期状态机
- 按账号/按平台限流
- 关闭流程收尾

不适合直接照搬：

- 其多平台抽象较重，本项目暂时只服务 B 站
- 连接解析层与录制层的组织方式可借鉴，但没必要完全复制目录结构

### 2.3 `bilibili-danmaku-client` / `bilibili-live-danmaku`

仓库特征：

- 主要是 B 站弹幕协议客户端/库
- 重点是 WebSocket 协议、事件解析、room id、心跳、压缩包解码

可借鉴点：

- **协议解析独立化**：WebSocket 包、op 码、压缩协议、心跳处理分层清楚
- **消息回调**：把解析结果以事件形式向上抛，而不是和业务强耦合
- **连接状态**：open / close / error 语义明确

适合迁移到本项目的模式：

- 把 `BilibiliRecorder` 里的协议解析单独抽出来
- 把 `ProcessPacket` / `CreateRecordedEvent` 作为纯协议层保留

不适合直接照搬：

- 它们通常不关心服务端持久化、归档、后台管理

## 4. 对本项目最有价值的迁移模式

- **统一账号选择入口**
- **账号健康状态模型**
- **generation / 版本号防过时回写**

- WebSocket 协议解析

- JSONL meta 行格式
- WBI / bili_ticket / cookie 刷新逻辑

### 4.3 不建议照搬的模式

- 连接池式单客户端多房间设计
- 分布式 runtime 协同
- 过重的任务优先级队列

## 5. 推荐的重构边界

### 第一层：协议层

- 负责 WebSocket 帧、压缩、心跳、消息解析
- 目标：纯函数化、可单测

### 第二层：账号与调度层

- 负责账号池、失败、冷却、分配、限流
- 目标：统一入口，不再散落到 Recorder / Service / Controller

### 第三层：录制与归档层

- 负责 session 创建、tmp 写入、final 归档、恢复
- 目标：jsonl 是事实源，tmp 只做恢复缓冲

### 第四层：查询与 API 层

- 负责前端接口、DTO、分页、统计
- 目标：对前端稳定输出 snake_case

### 8.1 `https://danmakus.com/`

公开线索：

- 首页显示：
  - 收录直播间数
  - 直播中数量
  - 片段数量
  - 本站速率 / 客户端速率 / 在线客户端
- 首页和 `client-doc` 明确说明：
  - 由于 B 站风控，站点本身无法获取完整弹幕
  - 提供 **Tauri 桌面客户端**
  - 客户端用于**采集直播间弹幕并上传到本站**
  - 支持 **多直播间同时采集**
  - 有 **账户页** 和“安全的账号认证机制”
- 直接抓到首页 HTML：
  - `div#app`
  - `/assets/index-*.js`
  - `/assets/vue-router-*.js`
  - `naive-ui` 相关 chunk
  - Cloudflare Pages Analytics

可推测架构：

- **核心不是中心化录制全部主播**，而是 **分布式客户端协作采集**
- 中心站点更像：
  - 元数据与查询平台
  - 聚合上传结果
  - 展示排行 / 搜索 / 片段
- 多账号调度方式很可能不是“中心服务器统一调度大量账号”，而是：
  - **每个客户端本地登录自己的 B 站账号**
  - 按用户意愿录制若干直播间
  - 将采集结果上传中心站点
- 从公开资源和外部技术线索看，主站前端高概率是 **Vue 3 + Vite + Naive UI**，站点本身偏静态前端 + 独立上传/聚合后端架构
- 存在进一步线索显示其可能使用 **REST + SignalR/WebSocket** 做状态同步，但这部分属于中等置信度推断

对本项目的启发：

- 如果未来 B 站风控继续收紧，**中心化单后端采集**会越来越难
- 除了“服务端多账号录制”方案，还可以考虑：
  - 官方后端录制
  - 辅助桌面客户端协作上传
  - 二者并存

特别值得借鉴：

- **客户端协作采集** 模式
- **站点速率 vs 客户端速率** 的可观测性
- 在架构上承认“平台风控约束”，而不是假设服务端永远能全量抓
- **主站展示层与采集层分离**

## 9. 从这些站点归纳出的共同模式

### 9.1 中心化全抓越来越难

`danmakus.com` 已直接公开承认：

- 平台风控使站点难以直接获取完整弹幕
- 需要依赖客户端协助录制

这说明一个现实：

> 如果你的目标是“功能健壮、支持多账号、长期稳定”，就不能假设中心化服务端一定能一直全量抓到所有直播间弹幕。

### 9.2 统计站和录制站通常不是同一个重点

观察这些站点：

- 有些偏实时录制/采集
- 有些偏索引/查询
- 有些偏专题展示

这提示你重构时最好分清：

- **录制核心**
- **索引与查询**
- **可视化展示**

不要把三者继续揉进一个大 Service / 大 Controller。

### 9.3 高频状态适合 websocket，不适合纯轮询

`vtb.cat` 明确提到把 status 接口改成 websocket。

这很适合你未来的管理后台：

- 录制器状态
- 在线速率
- 账号健康状态
- 错误告警

都适合做成 websocket 推送，而不是每几秒轮询一次 REST。

### 9.4 高成本数据必须异步化、分片化

`vtb.cat` 的更新日志说明：

- 多线程刷新粉丝
- 异步获取大航海数据
- 优化分区刷新逻辑

这说明：

> 大量 B 站用户/舰长/粉丝/分区数据，不能用“同步扫一遍”的思路做。

你未来要重构时也应把：

- 弹幕录制
- 用户统计
- 舰长/粉丝补全
- 排行/报表刷新

拆成不同异步任务流。

### 9.5 展示层与采集层几乎都在分离

从这些站点能看到一个共性：

- 公开站点负责展示和查询
- 采集、上传、索引、刷新通常在站点背后独立运行

这说明你新系统不应再把：

- 录制器
- 查询接口
- 管理后台
- 统计补全任务

全部塞进一个大服务里用同一种运行模型解决。

## 10. 对本项目重构的直接建议

结合现有开源项目和现有站点，最稳妥的目标架构是：

### 10.1 录制核心保持中心化

- 服务端负责主录制
- 多账号池负责房间分配
- `tmp → final` 归档以 jsonl 为核心

### 10.2 查询与索引独立出来

- 历史 jsonl 不再只是“播放文件”
- 应成为后续索引、统计、点歌提取、搜索的输入源

### 10.3 保留未来的客户端协作扩展能力

即使第一期不做，也建议在新系统设计时预留：

- 外部客户端上传 jsonl 或事件流
- 服务端接收补录数据
- 站点速率与客户端速率分开观测

### 10.4 管理后台改为 websocket + REST 混合

- CRUD：REST
- 状态/速率/告警：WebSocket

这会比现在单纯轮询更接近成熟数据站模式。

- 这是一个**实时弹幕机 / 控制台 / OBS 输出服务**
- 它和开放平台形成搭配：
  - 开放平台提供能力接口
  - Chat 提供实时接入和展示
- 更像一个完整的**工具链**而不是单点页面

### 11.3 对本项目的启发

- 录制系统未来可以拆成：
  - 录制核心
  - 实时输出/控制台
  - 索引/查询
  - 开放 API
- 如果做对外能力，建议把它们做成独立层，不要塞进一个 Controller 或一个大 Service

### 11.4 生态判断

LAPLACE 生态整体更像：

> **开放平台 + 实时工具 + 专用展示站 + 外部协作客户端**

这和你现在想做的“可重建、可扩展、多账号弹幕录制系统”很接近。
