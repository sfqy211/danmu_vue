### 2.2 P0 问题细节

#### P0-1: WBI 签名未实现

**位置**：`BilibiliService.cs` 全局

**影响**：B 站已对部分 Web API 强制要求 WBI 签名（`getDanmuInfo` 自 2025.5.26 起强制，`get_info` 等逐步收紧）。当前仍可工作的原因是指标尚未覆盖所有端点（`room_init` 暂未强制），但随时可能全量生效。所有 Web API 调用均缺少 `w_rid` + `wts` 参数，属于**预防性修复**。

**受影响的 API (按 WBI 强制程度)**：
| API | WBI 状态 | 调用频率 | 备注 |
|-----|----------|----------|------|
| `getDanmuInfo` | **已强制** (2025.5.26) | 每次连接 | 当前因其他风控豁免暂可用，随时可能封 |
| `room/v1/Room/get_info` | **已强制** | 高频 | |
| `x/web-interface/card` | **已强制** | 按需 | |
| `x/relation/stat` | **已强制** | 按需 | |
| `room/v1/Room/room_init` | 暂未强制 | 高频 | 预防性实现 |

**修复方式**：

1. 在 `BilibiliService` 中新增 `WbiSigner` 内部类
2. 所有 GET 请求自动检测并附加 `w_rid` + `wts`
3. 从 `api.bilibili.com/x/web-interface/nav` 获取 `img_key`/`sub_key`，缓存 60 分钟
4. **关键**：WBI 接口不设置 `Referer` header

### 2.3 P0+ 问题细节

#### P0+-1: Cookie 自动刷新缺失

**位置**：`BiliAccountService.cs`

**影响**：当前只有 24 小时定时刷新循环（使用 TV 端 `oauth2/refresh_token`），但 Web 端 Cookie（SESSDATA）会随敏感接口访问逐渐失效。一旦 Cookie 过期，账号立即被标记为异常并触发故障转移，但**没有自动恢复能力**。对于长期运行的录制后台，必须实现 Web Cookie 自动刷新。

**修复**：新增 `RefreshWebCookieAsync(uid)` 方法，实现完整的 Web Cookie 刷新流程（CorrespondPath → refresh_csrf → cookie/refresh → confirm/refresh）。

#### P0+-2: bili_ticket 风控票据缺失

**位置**：`BilibiliService.cs`

**影响**：开发指南将 `bili_ticket` 标记为 ⭐⭐⭐⭐ 风控关键。缺少 `bili_ticket` 是第一层风控拦截的触发条件之一。当前所有请求均未携带 `bili_ticket`。

**修复**：新增 `GetBiliTicketAsync()` 方法，通过 HMAC-SHA256 签名获取 `bili_ticket`，缓存约 3 天，过期前 1 小时自动刷新。

#### P0+-3: 全局频率控制缺失

**位置**：`LiveStatusService.cs`、`BilibiliService.cs`

**影响**：当前 `LiveStatusService` 使用 `SemaphoreSlim(10)` 控制并发，但没有全局频率限制。当监控房间数量增多时，总请求频率可能超过 B 站风控红线（单 IP 每分钟不超过 30 次请求）。

**修复**：新增 `BiliRateLimiter` 单例服务，实现三层频率控制（全局限流 → 请求去重+抖动 → 应用层调度）。

## 3. 重构目标

### 3.1 核心目标

1. **防封**：实现 WBI 签名，防止 B站 API 限流/封禁
2. **补数据**：补充弹幕消息中的勋章、大航海等用户身份信息
3. **高可用**：关键状态（\_roomAccountMap、LiveStatus）持久化到 Redis，重启不丢失
4. **可观测**：增加录制器心跳 + 健康巡检，异常自动告警

### 3.2 不做的事情

- **不拆分服务文件**：`BilibiliService.cs`（~200 行）、`BiliAccountService.cs`（~400 行）复杂度可控，合入 WBI 签名即可，不拆分子目录
- **不引入真正的 PM2**：当前 Docker 部署足够，只做命名规范化
- **Redis live:status 达到 P3**：`ProcessManager.GetProcesses()` 已覆盖录制中房间，全量状态缓存收益有限

## 4. 详细方案

### 4.1 P0: WBI 签名

**修改文件**：`BilibiliService.cs`

**新增内部类 `WbiSigner`**：

```
WbiSigner
├── MIXIN_KEY_ENC_TAB (static, 64 个 int)
├── GetMixinKey(imgKey + subKey) → 前 32 位
├── SignParams(params) → 添加 wts + 过滤 + 排序 + URL 编码 + MD5 → w_rid
├── GetWbiKeysAsync() → 调用 nav 接口，缓存 60 分钟
├── SanitizeValue(v) → 过滤 !'()* 字符
└── CreateWbiRequest(url, params, cookie?) → 构造带签名的 HttpRequestMessage
```

**签名算法（参照 bilibili-api Python 实现精确还原）**：

```
步骤 1: 获取 mixin_key
  GET https://api.bilibili.com/x/web-interface/nav
  响应 data.wbi_img.img_url → 取文件名部分作为 img_key
  响应 data.wbi_img.sub_url → 取文件名部分作为 sub_key
  例: img_url = "https://i0.hdslb.com/bfs/wbi/7eba08db25c46246b18c4d7a3a2f9c4b.png"
      → img_key = "7eba08db25c46246b18c4d7a3a2f9c4b"
  拼接 rawKey = img_key + sub_key
  用 MIXIN_KEY_ENC_TAB 重排: result = new string(MIXIN_KEY_ENC_TAB.Select(i => rawKey[i]))
  取前 32 字符作为 mixin_key
  缓存 mixin_key 60 分钟，过期后重新获取

步骤 2: 签名参数
  params["wts"] = 当前 Unix 秒级时间戳
  对每个 value 调用 SanitizeValue: 移除 ! ' ( ) * 共 5 个字符
  按 key 字典序排序所有参数
  URL 编码: key=value&key2=value2 (使用 Uri.EscapeDataString, 空格→%20)
  拼接: encodedParams + mixin_key
  MD5 哈希: w_rid = MD5(concatenated).ToLowerInvariant()
  将 w_rid 添加到 params

步骤 3: 构造请求
  最终 URL = baseUrl + "?" + 编码后的参数字符串
  **WBI 签名接口不设置 Referer header** (与当前全局 Referer 冲突，需显式移除)
  设置 Cookie (如有)、User-Agent、buvid headers
```

**MIXIN_KEY_ENC_TAB 常量**（64 个索引，与 Python 实现完全一致）：

```csharp
private static readonly int[] MixinKeyEncTab = {
    46, 47, 18, 2, 53, 8, 23, 32, 15, 50, 10, 31, 58, 3, 45, 35, 27, 43, 5, 49,
    33, 9, 42, 19, 29, 28, 14, 39, 12, 38, 41, 13, 37, 48, 7, 16, 24, 55, 40, 61,
    26, 17, 0, 1, 60, 51, 30, 4, 22, 25, 54, 21, 56, 59, 6, 63, 57, 62, 11, 36, 20, 34, 44, 52
};
```

**SanitizeValue 实现**：

```csharp
private static string SanitizeValue(string value)
{
    // 移除 ! ' ( ) * 共 5 个字符，与 B 站 WBI 签名要求一致
    var sb = new StringBuilder(value.Length);
    foreach (var c in value)
    {
        if (c != '!' && c != '\'' && c != '(' && c != ')' && c != '*')
            sb.Append(c);
    }
    return sb.ToString();
}
```

**WBI 签名重试机制**（参照 Python 实现的 -403 重试）：

```csharp
// 当 API 返回 code == -403 时，清除缓存的 mixin_key 并重试
// 最多重试 3 次
if (responseCode == -403 && retryCount < MaxWbiRetry)
{
    _wbiMixinKey = null; // 强制重新获取
    retryCount++;
    continue;
}
```

**注意事项**：

- `Uri.EscapeDataString` 将空格编码为 `%20`（正确），不要使用 `WebUtility.UrlEncode`（编码为 `+`）
- **WBI 接口不设置 Referer header**：当前 `BilibiliService` 构造函数全局设置了 `Referer: https://www.bilibili.com`，WBI 请求必须显式移除或使用独立 HttpRequestMessage
- `img_key`/`sub_key` 缓存 60 分钟，过期后重新从 `nav` 接口获取
- -403 响应码表示 WBI 签名过期，需清除缓存并重试

### 4.2 P0+: Cookie 自动刷新

**修改文件**：`BiliAccountService.cs`

**背景**：当前只有 24 小时定时刷新循环（`StartAutoRefreshLoop`），使用 TV 端 `oauth2/refresh_token` 接口。但 Web 端 Cookie（SESSDATA）会随敏感接口访问逐渐失效，需要独立的 Web Cookie 刷新机制。

**参照 bilibili-api Python 实现的 Cookie 刷新流程**：

```
CookieRefreshService
├── CheckCookieHealthAsync(uid) → 调用 nav 接口检查 isLogin
├── RefreshWebCookieAsync(uid) → 完整的 Web Cookie 刷新流程
│   ├── Step 1: 生成 CorrespondPath (RSA-OAEP 加密)
│   ├── Step 2: GET /correspond/1/{correspondPath} 提取 refresh_csrf
│   ├── Step 3: POST /x/passport-login/web/cookie/refresh
│   └── Step 4: POST /x/passport-login/web/confirm/refresh
├── _correspondPathCache → 缓存 correspondPath (有效期短)
└── _refreshTimer → 定时检查并刷新 (每 6 小时)
```

**详细刷新流程**：

```
Step 1: 生成 CorrespondPath
  获取 RSA 公钥: GET https://api.bilibili.com/x/web-interface/nav
    → data.wbi_img.img_url (复用 WBI 的 nav 接口)
  实际上 CorrespondPath 使用的是 passport 的 RSA 公钥:
    GET https://passport.bilibili.com/x/passport-login/web/key
    → data.hash (拼接在明文前)
    → data.key (PEM 格式 RSA 公钥)
  明文 = hash + "refresh_{timestamp}"
  使用 RSA-OAEP 加密 → Base64 编码 → URL 编码 → correspondPath

Step 2: 提取 refresh_csrf
  GET https://www.bilibili.com/correspond/1/{correspondPath}
    → HTML 响应中查找 <div id="1-name">{refresh_csrf}</div>
  需要携带当前 SESSDATA Cookie

Step 3: 刷新 Cookie
  POST https://passport.bilibili.com/x/passport-login/web/cookie/refresh
  Body: csrf={bili_jct}&refresh_csrf={refresh_csrf}&refresh_token={ac_time_value}&source=main_web
  Headers: Cookie (当前 SESSDATA 等), Referer: https://www.bilibili.com
  响应:
    → 新的 SESSDATA, bili_jct (在 Set-Cookie 中)
    → 新的 refresh_token (在响应体中)
    → timestamp (用于 Step 4)

Step 4: 确认刷新
  POST https://passport.bilibili.com/x/passport-login/web/confirm/refresh
  Body: csrf={new_bili_jct}&refresh_token={new_refresh_token}&source=main_web
  此步骤确认刷新，使旧 Cookie 正式失效
```

**与现有代码的整合**：

```csharp
// BiliAccountService.cs 新增方法
public async Task<bool> RefreshWebCookieAsync(long uid)
{
    // 1. 获取账号当前 Cookie
    var account = await GetAsync(uid);
    if (account == null) return false;
    var cookieDict = JsonSerializer.Deserialize<Dictionary<string, string>>(account.CookieJson);

    // 2. 检查是否需要刷新 (SESSDATA 即将过期或已失效)
    if (account.ExpiresAt.HasValue && account.ExpiresAt.Value > DateTime.UtcNow.AddHours(6))
        return true; // 还没过期，不需要刷新

    // 3. 获取 RSA 公钥
    var (hash, publicKey) = await GetPassportRsaKeyAsync();

    // 4. 生成 CorrespondPath
    var correspondPath = GenerateCorrespondPath(hash, publicKey);

    // 5. 提取 refresh_csrf
    var refreshCsrf = await ExtractRefreshCsrfAsync(correspondPath, BuildCookieString(account.CookieJson));
    if (refreshCsrf == null) return false;

    // 6. 刷新 Cookie
    var (newCookieDict, newRefreshToken) = await PostCookieRefreshAsync(
        cookieDict!["bili_jct"], refreshCsrf, cookieDict!["refresh_token"],
        BuildCookieString(account.CookieJson));

    // 7. 确认刷新
    await PostConfirmRefreshAsync(newCookieDict["bili_jct"], newRefreshToken);

    // 8. 更新数据库
    await UpdateAccountCookieAsync(uid, newCookieDict, newRefreshToken);
    return true;
}
```

**定时刷新策略**：


```csharp
private static string EncryptWithPublicKey(string plaintext, string pemKey)
{
    using var rsa = RSA.Create();
    rsa.ImportFromPem(pemKey);
    var encrypted = rsa.Encrypt(Encoding.UTF8.GetBytes(plaintext), RSAEncryptionPadding.OaepSHA1);
    return Convert.ToBase64String(encrypted);
}

private string GenerateCorrespondPath(string hash, string publicKey)
{
    var timestamp = DateTimeOffset.UtcNow.ToUnixTimeSeconds();
    var plaintext = $"{hash}refresh_{timestamp}";
    var encrypted = EncryptWithPublicKey(plaintext, publicKey);
    return Uri.EscapeDataString(encrypted);
}
```

### 4.3 P0+: bili_ticket 风控票据

**修改文件**：`BilibiliService.cs`

**背景**：开发指南将 `bili_ticket` 标记为 ⭐⭐⭐⭐ 风控关键。缺少 `bili_ticket` 是第一层风控拦截的触发条件之一。

**参照 bilibili-api Python 实现的 bili_ticket 获取流程**：

```
BiliTicketService (作为 BilibiliService 的内部组件)
├── _biliTicket: string? (缓存)
├── _biliTicketExpires: long (过期时间戳)
├── GetBiliTicketAsync() → 获取或刷新 bili_ticket
├── GenerateBiliTicketSign() → HMAC-SHA256 签名
└── InvalidateTicket() → 强制刷新 (WBI -403 时调用)
```

**获取流程**：

```
Step 1: 生成签名
  key = "XgwSnGZ1p" (固定密钥)
  message = "ts{timestamp}" (当前秒级时间戳)
  hexsign = HMAC-SHA256(key, message).ToLowerInvariant()

Step 2: 请求票据
  POST https://api.bilibili.com/bapis/bilibili.api.ticket.v1.Ticket/GenWebTicket
  Body (form-urlencoded):
    key_id=ec02
    hexsign={hexsign}
    context[ts]={timestamp}
    csrf= (空字符串，无需登录)
  Headers:
    User-Agent: (标准浏览器 UA)
    Content-Type: application/x-www-form-urlencoded

Step 3: 解析响应
  response.data.ticket → bili_ticket 字符串 (有效期约 3 天)
  response.data.created_at → 创建时间
  response.data.ttl → 有效秒数

Step 4: 缓存与刷新
  缓存 bili_ticket 和过期时间
  过期前 1 小时自动刷新
  WBI 返回 -403 时强制刷新
```

**.NET 实现**：

```csharp
// BilibiliService.cs 内部方法
private string? _biliTicket;
private long _biliTicketExpires;
private readonly object _ticketLock = new();

private async Task<string> GetBiliTicketAsync()
{
    lock (_ticketLock)
    {
        if (_biliTicket != null && DateTimeOffset.UtcNow.ToUnixTimeSeconds() < _biliTicketExpires - 3600)
            return _biliTicket;
    }

    var timestamp = DateTimeOffset.UtcNow.ToUnixTimeSeconds();
    var hexsign = GenerateBiliTicketSign(timestamp);

    var content = new FormUrlEncodedContent(new Dictionary<string, string>
    {
        ["key_id"] = "ec02",
        ["hexsign"] = hexsign,
        ["context[ts]"] = timestamp.ToString(),
        ["csrf"] = ""
    });

    var response = await _httpClient.PostAsync(
        "https://api.bilibili.com/bapis/bilibili.api.ticket.v1.Ticket/GenWebTicket", content);
    var json = await response.Content.ReadAsStringAsync();
    using var doc = JsonDocument.Parse(json);
    var data = doc.RootElement.GetProperty("data");

    var ticket = data.GetProperty("ticket").GetString()!;
    var ttl = data.GetProperty("ttl").GetInt64();

    lock (_ticketLock)
    {
        _biliTicket = ticket;
        _biliTicketExpires = DateTimeOffset.UtcNow.ToUnixTimeSeconds() + ttl;
    }

    return ticket;
}

private static string GenerateBiliTicketSign(long timestamp)
{
    var key = "XgwSnGZ1p";
    var message = $"ts{timestamp}";
    using var hmac = new HMACSHA256(Encoding.UTF8.GetBytes(key));
    var hash = hmac.ComputeHash(Encoding.UTF8.GetBytes(message));
    return Convert.ToHexString(hash).ToLowerInvariant();
}
```

**HTTP Header 注入**：

```
所有 BilibiliService 发出的请求都应携带:
  buvid3: {生成或缓存的 buvid3}
  buvid4: {生成或缓存的 buvid4}
  b_nut: {随机 6 位字母数字}
  b_lsid: {timestamp}_{随机16进制}
  bili_ticket: {从 GetBiliTicketAsync() 获取}
```

**注意事项**：

- `bili_ticket` 有效期约 3 天（259200 秒），提前 1 小时刷新
- `key_id=ec02` 和 `key=XgwSnGZ1p` 是 B 站 Web 端固定值
- 此接口不需要 Cookie，但需要正确的 HMAC 签名
- WBI 返回 -403 时应同时刷新 `bili_ticket` 和 `wbi_keys`

### 4.4 P0+: 三层频率控制

**修改文件**：新增 `Services/BiliRateLimiter.cs`，修改 `LiveStatusService.cs`、`BilibiliService.cs`

**背景**：当前 `LiveStatusService` 使用 `SemaphoreSlim(10)` 控制并发，但没有全局频率限制。当监控房间数量增多时，总请求频率可能超过 B 站风控红线（单 IP 每分钟不超过 30 次）。

**参照 bililive-go 的三层防护架构，简化为单平台（Bilibili）版本**：

```
┌─────────────────────────────────────────────────────────────────┐
│  Layer 1: 全局平台限流器 (BiliRateLimiter)                      │
│  • 单例，所有 Bilibili API 请求共享                             │
│  • 最小请求间隔: 500ms (可配置)                                  │
│  • WaitForAsync() 异步等待直到满足间隔                           │
│  • 无论多少房间/服务，全局不超过频率限制                          │
├─────────────────────────────────────────────────────────────────┤
│  Layer 2: 请求去重 + 随机抖动 (LiveStatusService)               │
│  • 多调用方共享同一次请求结果 (ConcurrentDictionary 缓存)        │
│  • 随机抖动 -3s ~ +3s，避免同时请求                              │
│  • 可配置轮询间隔 (默认 120s)                                    │
├─────────────────────────────────────────────────────────────────┤
│  Layer 3: 应用层调度 (各服务独立)                                │
│  • LiveStatusService: 每房间独立调度周期                         │
│  • VupInfoScheduler: 每 30 分钟更新一个 VUP                      │
│  • BilibiliRecorder: WebSocket 心跳独立                          │
│  • 无全局 SemaphoreSlim，靠 Layer 1 自然限速                    │
└─────────────────────────────────────────────────────────────────┘
```

**Layer 1: BiliRateLimiter 实现**：

```csharp
public class BiliRateLimiter
{
    private readonly TimeSpan _minInterval;
    private DateTime _lastAccess = DateTime.MinValue;
    private readonly SemaphoreSlim _sync = new(1, 1);
    private readonly ILogger<BiliRateLimiter> _logger;

    public BiliRateLimiter(ILogger<BiliRateLimiter> logger, TimeSpan? minInterval = null)
    {
        _logger = logger;
        _minInterval = minInterval ?? TimeSpan.FromMilliseconds(500); // 默认 500ms
    }

    /// <summary>
    /// 等待直到满足最小间隔后放行。支持 CancellationToken。
    /// 
    /// 设计意图：控制请求发起频率（每 500ms 最多发起一个），而非控制并发数。
    /// 调用方在 WaitForAsync 返回后发起 HTTP 请求，此时信号量已释放，
    /// 因此多个请求可能同时在途——这是预期行为：慢请求不应阻塞后续请求的调度。
    /// 如需限制同一时刻最多一个请求在途，应将 HTTP 调用放在 _sync 保护范围内，
    /// 但那会导致慢请求（如网络超时）阻塞整个系统，不适合本场景。
    /// </summary>
    public async Task WaitForAsync(CancellationToken ct = default)
    {
        await _sync.WaitAsync(ct);
        try
        {
            var elapsed = DateTime.UtcNow - _lastAccess;
            if (elapsed < _minInterval)
            {
                var wait = _minInterval - elapsed;
                _logger.LogDebug("Rate limit: waiting {Ms}ms", (int)wait.TotalMilliseconds);
                await Task.Delay(wait, ct);
            }
            _lastAccess = DateTime.UtcNow;
        }
        finally
        {
            _sync.Release();
        }
    }

    /// <summary>
    /// 强制放行（用于紧急请求，如录制器启动）
    /// </summary>
    public void ForceAccess()
    {
        lock (this) { _lastAccess = DateTime.UtcNow; }
    }
}
```

**注册为单例**：


### 4.8 P1: Redis 增量 Dump（原 4.5）

**修改文件**：`BilibiliRecorder.cs`

**增量 Dump 机制**：

```
每 5 分钟触发一次增量 dump:
  1. 读取当前 Redis List 长度 → totalCount
  2. 读取 dumpOffset (上次已持久化的位置)
  3. 从 dumpOffset 开始 LRANGE 新条目
  4. 追加到 JSONL 文件
  5. 更新 dumpOffset = totalCount (写入 Redis meta:dumpOffset)
  6. 直播结束时做最终全量 dump，然后清理 meta:dumpOffset
```

**崩溃恢复流程**：

当 `ProcessManager.RestoreRecordersAsync()` 恢复录制器时，录制器检查 `meta:dumpOffset`：

```
if Redis 存在 meta:dumpOffset:
  说明这是崩溃恢复场景，上次会话的弹幕仍在 Redis 中
  1. 从 dumpOffset 开始将剩余 Redis 数据追加到 JSONL
  2. 标记 JSONL 可用于 DanmakuProcessor 处理
  3. 清理 meta:dumpOffset
else:
  正常新会话
```

这样即使进程崩溃，最多丢失 5 分钟的弹幕数据，且恢复后自动补全。

**与现有会话恢复的整合**：

`ProcessManager.RestoreRecordersAsync` 当前只检查 `auto_record=1` 的房间并重新启动录制器。重构后，每个恢复的 `BilibiliRecorder` 在 `StartAsync` 时自动检测 `meta:dumpOffset` 并完成数据补全，无需修改 `ProcessManager`。

## 附录 A: 未纳入本次重构的问题

以下问题经评估后决定**不纳入本次重构**：

| 问题                                                       | 原因                                                             |
| ---------------------------------------------------------- | ---------------------------------------------------------------- |
| 拆分为 BilibiliApiClient/AuthManager/CacheService 三个文件 | BilibiliService.cs 仅 ~200 行，复杂度可控，非必要不拆分          |
| Redis live:status 全局状态缓存                             | ProcessManager.GetProcesses() 已覆盖录制中房间，全量状态收益有限 |
| Redis up:profile 缓存                                      | DB rooms 表 + VupInfoScheduler 已覆盖                            |
| 真正的 PM2 进程管理                                        | 当前 Docker 部署足够，无需额外进程管理器                         |
| 多账号负载均衡策略                                         | Round-robin 对当前规模足够                                       |
| 并行启动 RestoreRecordersAsync                             | 当前串行 + 5 秒间隔是风控需要，不能并行                          |
