# danmu_vue 重构方案（基于当前代码现状）

> 本文档只保留**当前项目已经存在**的实现、接口、模块边界和可直接落地的重构方向。  
> 未在当前仓库实现的 PostgreSQL / WebSocket / 上传开放接口 / `server_net_v2` 目录方案已全部删除。

---

## 0. 文档目标

这份文档用于在真正开始重构前，先把当前项目的现实情况说清楚：

1. 当前代码到底有哪些接口
2. 前端实际依赖了哪些接口
3. 后端当前有哪些核心服务与耦合点
4. 现在最适合优先拆分和治理的部分是什么

本文档**不是未来架构蓝图**，而是当前代码审查与重构落点说明。

## 1. 当前项目事实

### 1.1 技术栈

当前项目实际使用：

| 层 | 技术 | 状态 |
|---|---|---|
| 前端 | Vue 3 + TypeScript + Vite + Pinia + Element Plus | 已使用 |
| 后端 | .NET 9 + ASP.NET Core | 已使用 |
| 数据库 | MySQL + EF Core | 已使用 |
| 缓存/热缓冲 | Microsoft Garnet（Redis 协议） | 已使用 |
| 历史归档 | JSONL / XML | 已使用 |
| 日志实时流 | SSE | 已使用 |

### 1.2 当前运行模型

当前代码不是单一数据库系统，而是三套状态源并存：

1. **MySQL**
   - 房间配置 `rooms`
   - 场次记录 `sessions`
   - 点歌记录 `song_requests`
   - B站账号 `bili_accounts`
   - 更新日志 `changelog_entries`

2. **Redis / Garnet**
   - 录制中的弹幕队列
   - live session key
   - recorder heartbeat
   - live status cache
   - 账号分配状态

3. **文件系统**
   - `server/data/danmaku/`：历史归档 JSONL / XML
   - `server/data/danmaku_tmp/`：临时录制文件 / 恢复文件

### 1.3 当前核心问题

- 查询链路仍大量依赖 JSONL 文件读取
- 管理端接口高度集中在 `AdminController.cs`
- `DanmakuService.cs` 职责过多
- `BilibiliRecorder` / `ProcessManager` / `DanmakuService` 通过事件与委托隐式耦合

## 5. 当前结构概览

### 5.1 Controllers

- `DanmakuController.cs`：公开查询接口
- `AdminController.cs`：管理端全集中
- `ChangelogController.cs`：公开更新日志

### 5.2 Services

- `ProcessManager.cs`：录制器实例生命周期
- `BilibiliRecorder.cs`：WebSocket 录制、心跳、Redis 写入
- `DanmakuService.cs`：场次、文件、Redis、DB、导入转换
- `BiliAccountService.cs`：账号池、Cookie、二维码登录、状态管理
- `LiveStatusService.cs`：轮询房间直播状态
- `HealthCheckService.cs`：检查心跳和状态漂移

### 5.3 当前耦合点

- `BilibiliRecorder -> ProcessManager -> DanmakuService` 通过事件委托串联
- `JSONL / Redis / DB` 三套状态并存
- `AdminController` 单文件过大
- 前端归一化逻辑较多

---

## 6. 当前可直接落地的重构优先级

### P0

- 冻结现有接口清单
- 清理无效接口引用（如 `/files`）
- 保证自动重启与幂等启动逻辑稳定
- 保证录制开始时间口径稳定

### P1

- 拆 `AdminController.cs`
- 拆 `DanmakuService.cs`
- 拆 `AdminView.vue`
- 按资源域整理前端 API 文件

### P2

- 收敛字段命名
- 逐步减少前端兼容层

---

## 7. 当前重构的约束与风险

- 不能直接删掉 JSONL / Tmp 恢复链路
- 不能直接假设字段命名已经统一
- 不能在一次改动里同时改接口、改存储、改前端页面结构
- 不能只写理想化未来方案而跳过现有代码迁移成本

---

## 8. 结论

当前项目最需要的不是“先设计全新架构”，而是：

1. 先把现有接口和依赖关系收清楚
2. 先把超大控制器/服务拆开
3. 先把前后端接口清单完整固定下来
4. 在不破坏 MySQL / Redis / JSONL 现有运行链路的前提下逐步重构
