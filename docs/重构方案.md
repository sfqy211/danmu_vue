# danmu_vue 重构方案（高查询量最终版）

> 目标：在接受较高开发成本的前提下，重建一个**性能优先、结构清晰、多账号健壮、可持续演进**的弹幕录制与查询系统。\
> 定位：不是对现有后端做修补，而是基于现有 JSONL 历史资产与可修改前端的前提，做一次**受控的拆骨重建**。

---

## 0. 执行摘要

### 0.1 结论

- **后端架构需要重构**，但不需要推翻 .NET 技术栈。
- **主数据库必须切换为 PostgreSQL**，用于高并发查询、全文搜索、统计聚合与索引。
- **Redis/Garnet 保留**，作为录制热缓冲、状态缓存与临时映射。
- **历史 JSONL 保留**，但仅作为归档、导出和审计格式，不再作为查询主路径。
- **前端接口可以重定义**，不再兼容旧 MySQL 数据模型。
- **多账号能力必须成为系统一等公民**，不是在单账号逻辑上打补丁。

### 0.2 最终推荐技术栈

| 层 | 技术 | 说明 |
|---|---|---|
| 前端 | Vue 3 + TypeScript + Vite + Pinia + Element Plus | 保留 |
| 后端 | .NET 9 + ASP.NET Core | 保留 |
| 主数据库 | **PostgreSQL 16** | 替换 MySQL，承担查询、索引、统计、配置 |
| 热缓冲 | Microsoft Garnet（Redis 协议） | 保留，用于录制热缓冲、状态缓存 |
| 历史归档 | JSONL | 保留，但不再作为常规查询主源 |
| 实时推送 | WebSocket | 新增，替代关键状态轮询 |
| 部署 | Docker + 1Panel + COS/CDN | 保留，删除 MySQL 容器 |

## 1. 重构目标

### 1.1 核心目标

1. **高性能**：避免查询时全量读文件、全量扫内存。
2. **多账号健壮**：账号池、失败隔离、冷却、粘性分配、限流都要统一。
3. **查询可扩展**：数据库作为查询主路径，支持高查询量和全文检索。
4. **状态可观测**：录制状态、账号健康、在线速率、告警统一可推送。
5. **归档可靠**：`tmp → final` 幂等，crash 可恢复。
6. **接口稳定**：前端接口统一 snake_case 输出，减少兼容层。
7. **可扩展**：预留外部客户端协作上传能力。

### 1.2 不再做的事情

- 不再兼容旧 MySQL 数据库。
- 不再把 JSONL 当做随机查询主存。
- 不再让多个服务各自决定账号选择。
- 不再让后台状态主要靠 REST 轮询。

---

## 2. 当前问题概括

### 2.1 结构性问题

- 单账号演进到多账号后，账号选择逻辑分散在多个服务。
- `BilibiliService`、`BiliAccountService`、`BilibiliRecorder`、`ProcessManager` 之间通过 delegate/event 隐式耦合。
- 录制中数据写 Redis，最终又写 JSONL，再由文件监控导入数据库，链路过长。
- 查询侧对 JSONL 做全量加载 + 内存过滤，性能瓶颈明显。
- 管理后台字段命名和响应结构长期处于兼容状态。

### 2.2 性能瓶颈

- `GetDanmakuPagedAsync` 读整份文件后分页。
- `BuildAnalysis` / `BuildGiftAnalysis` 每次全量重算。
- `GetSessions` 在 C# 里 GroupBy 去重。
- `DanmakuProcessor` 对文件变更重新解析整文件。

          ▼                ▼                ▼
┌────────────────┐ ┌────────────────┐ ┌────────────────────┐
│ Recorder Core  │ │ Query / Index  │ │ Realtime / Control │
│                │ │                │ │                    │
│ - AccountPool  │ │ - SessionIndex │ │ - RealtimeHub      │

### 3.2 设计原则

- **PostgreSQL 是查询主库**。
- **JSONL 是归档事实源**。
- **Garnet 是热缓冲与状态缓存**。
- **账号选择统一入口**。
- **状态变更通过 WebSocket 推送**。

## 6. `tmp → final` 新规则

### 6.1 规则

- `tmp` 只用于录制中和恢复。
- `final` 是稳定归档结果。
- 归档必须幂等。
- 查询只默认读 `final`。

### 6.2 生命周期

1. 开播：创建 tmp session。
2. 录制中：增量写入 tmp / 热缓冲。
3. 正常结束：flush 热缓冲，生成 final。
4. 异常退出：保留 tmp，等待重启恢复。
5. 恢复时：如果 tmp 未完成且直播仍在线，继续写；否则 finalize。

### 6.3 不再保留的旧语义

- 不再依赖 DB 里是否已有 `redis:` 前缀来判断会话活跃。
- 不再把文件状态和数据库状态双向猜测。

## 7. 多账号设计

### 7.1 账号状态机

- `healthy`
- `cooling_down`
- `expired`
- `disabled`
- `unknown`

### 7.2 账号选择入口

统一只通过一个入口做账号决策：

- 房间上下文请求：sticky assignment + failover
- 通用请求：从健康池中选默认可用账号

### 7.3 限流

必须分两层：

- **账号级限流**
- **全局并发/速率保护**

### 7.4 账号状态持久化

- `room -> account` 映射建议放 PostgreSQL 或持续化 Redis
- `account failure` 冷却状态必须可恢复，不能只放内存

### 7.5 账号持久化字段定案

账号状态不能只靠内存维护，数据库必须保存：

- `status`
- `cooling_until`

这样系统重启后仍能恢复账号冷却窗口与健康状态。

## 8. 录制核心设计

### 8.1 模块拆分

#### AccountPool
- 分配账号
- 标记失败
- 冷却恢复
- 默认账号预热

#### RoomRuntimeManager
- 启停 recorder
- 维护 generation
- 记录运行时状态

#### BilibiliLiveConnector
- 获取 room_init
- 获取 danmu token / host
- 建立 WS/TCP
- 心跳与重连

#### DanmakuEventParser
- 协议解包
- 事件解析
- 产生统一事件模型

#### SessionArchiver
- tmp 写入
- final 封口
- 恢复合并

### 8.2 状态机

Recorder：

```text
idle -> starting -> waiting_live -> connecting -> recording -> finalizing -> stopped
```

Session：

```text
created_tmp -> writing_tmp -> ready_to_finalize -> finalized
```

### 8.3 generation 保护

每次启动/重连都带 generation：

- 旧异步结果回来的时候先比对 generation
- 不匹配则丢弃
- 避免旧状态覆盖新状态

## 14. 风险与约束

### 14.1 关键风险

- PostgreSQL 并发写入需要批量写策略。
- `tmp/final` 归档必须幂等。
- 旧 JSONL 历史数据若不导入 PostgreSQL，将无法在新系统统一查询。
- 多账号状态如果仍停留在内存里，重启后会漂移。

### 14.2 关键约束

- 不再兼容旧 MySQL 数据库。
- JSONL 只做归档和导出。
- 不允许查询时全量读文件。
- 不允许账号选择分散在多个模块。

---

## 15. 验收标准

1. 单次查询弹幕不再全量读文件。
2. 多账号切换、失败冷却、恢复都能工作。
3. 录制结束后 `final` 文件稳定生成。
4. 录制中断后可恢复，不丢末尾数据。
5. 后台状态可通过 WebSocket 实时更新。
6. 旧 MySQL 不再参与运行。
7. 前端接口字段统一，兼容层可逐步删除。

---

## 16. 结语

这次重构的核心不是“把代码写得更漂亮”，而是：

- 把**JSONL 归档**和**在线查询**分开
- 把**多账号调度**做成系统主链路
- 把**实时状态**从轮询改成推送
- 把**MySQL 的低性价比**替换为更适合高查询量场景的 PostgreSQL

如果按这个方案执行，新系统会从“单账号补丁式演进”变成“真正可持续的多账号录制与查询平台”。

## 17. PostgreSQL DDL（可执行 SQL）

> 说明：以下 DDL 是新系统第一版可直接落库的基线方案。\
> 目标：配置、索引、查询、统计全部统一在 PostgreSQL 中；JSONL 仅作为归档与导出格式保留。

```sql
-- =========================
-- Extensions
-- =========================
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- =========================
-- rooms
-- =========================
CREATE TABLE IF NOT EXISTS rooms (
    id              BIGSERIAL PRIMARY KEY,
    room_id         BIGINT NOT NULL,
    uid             TEXT NOT NULL,
    name            TEXT NOT NULL DEFAULT '',
    remark          TEXT,
    playlist_url    TEXT,
    auto_record     INTEGER NOT NULL DEFAULT 1,
    last_live_time  BIGINT NOT NULL DEFAULT 0,
    followers       BIGINT NOT NULL DEFAULT 0,
    guard_num       BIGINT NOT NULL DEFAULT 0,
    video_count     BIGINT NOT NULL DEFAULT 0,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT uq_rooms_room_id UNIQUE (room_id)
);

CREATE INDEX IF NOT EXISTS idx_rooms_uid ON rooms(uid);
CREATE INDEX IF NOT EXISTS idx_rooms_auto_record ON rooms(auto_record);

-- =========================
-- bili_accounts
-- =========================
CREATE TABLE IF NOT EXISTS bili_accounts (
    uid             BIGINT PRIMARY KEY,
    name            TEXT,
    face            TEXT,
    cookie_json     TEXT NOT NULL DEFAULT '',
    expires_at      TIMESTAMPTZ,
    status          TEXT NOT NULL DEFAULT 'unknown',
    cooling_until   TIMESTAMPTZ,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_bili_accounts_expires_at ON bili_accounts(expires_at);
CREATE INDEX IF NOT EXISTS idx_bili_accounts_status ON bili_accounts(status);
CREATE INDEX IF NOT EXISTS idx_bili_accounts_cooling_until ON bili_accounts(cooling_until);

-- =========================
-- sessions
-- =========================
CREATE TABLE IF NOT EXISTS sessions (
    id                  BIGSERIAL PRIMARY KEY,
    uid                 TEXT NOT NULL,
    room_id             BIGINT NOT NULL,
    room_title          TEXT NOT NULL DEFAULT '',
    user_name           TEXT NOT NULL DEFAULT '',
    start_time          BIGINT NOT NULL,
    end_time            BIGINT,
    status              TEXT NOT NULL DEFAULT 'created_tmp',
    tmp_path            TEXT,
    final_path          TEXT,
    summary_json        JSONB,
    gift_summary_json   JSONB,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT uq_sessions_uid_room_start UNIQUE (uid, room_id, start_time)
);

CREATE INDEX IF NOT EXISTS idx_sessions_room_start ON sessions(room_id, start_time DESC);
CREATE INDEX IF NOT EXISTS idx_sessions_uid_start ON sessions(uid, start_time DESC);
CREATE INDEX IF NOT EXISTS idx_sessions_status ON sessions(status);

-- =========================
-- danmaku_events (partitioned)
-- 说明：按月分区，便于高查询量下的索引维护、冷热数据管理和归档清理。
-- =========================
CREATE TABLE IF NOT EXISTS danmaku_events (
    id                  BIGSERIAL NOT NULL,
    session_id          BIGINT NOT NULL REFERENCES sessions(id) ON DELETE CASCADE,
    event_type          TEXT NOT NULL,
    timestamp           BIGINT NOT NULL,
    uid                 TEXT,
    user_name           TEXT,
    text                TEXT,
    text_jpn            TEXT,
    price               DOUBLE PRECISION,
    count               INTEGER NOT NULL DEFAULT 1,
    gift_name           TEXT,
    coin_type           TEXT,
    guard_level         INTEGER,
    medal_level         INTEGER,
    medal_name          TEXT,
    medal_anchor        TEXT,
    medal_room_id       INTEGER,
    medal_guard_level   INTEGER,
    medal_is_light      BOOLEAN,
    medal_anchor_uid    BIGINT,
    ul_level            INTEGER,
    wealth_level        INTEGER,
    duration            INTEGER,
    raw_command         TEXT,
    raw_json            JSONB,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (id, timestamp)
) PARTITION BY RANGE (timestamp);

-- 月分区示例：按需由后台任务预创建当前月 / 下月分区。
CREATE TABLE IF NOT EXISTS danmaku_events_2025_01
PARTITION OF danmaku_events
FOR VALUES FROM (1735689600000) TO (1738368000000);

CREATE INDEX IF NOT EXISTS idx_events_sess_time_2025_01
    ON danmaku_events_2025_01(session_id, timestamp);

CREATE INDEX IF NOT EXISTS idx_events_sess_type_2025_01
    ON danmaku_events_2025_01(session_id, event_type);

CREATE INDEX IF NOT EXISTS idx_events_sess_uid_2025_01
    ON danmaku_events_2025_01(session_id, uid);

CREATE INDEX IF NOT EXISTS idx_events_text_trgm_2025_01
    ON danmaku_events_2025_01 USING GIN (text gin_trgm_ops);

-- 分区管理说明：
-- 1. 由 Background/PartitionManagerService 在启动时检查并创建“当前月 + 下月”分区
-- 2. 每日巡检一次，确保未来分区存在
-- 3. 历史分区不自动删除，清理由单独的归档/运维任务控制

-- =========================
-- session_stats
-- =========================
CREATE TABLE IF NOT EXISTS session_stats (
    session_id          BIGINT PRIMARY KEY REFERENCES sessions(id) ON DELETE CASCADE,
    total_count         BIGINT NOT NULL DEFAULT 0,
    comment_count       BIGINT NOT NULL DEFAULT 0,
    gift_count          BIGINT NOT NULL DEFAULT 0,
    sc_count            BIGINT NOT NULL DEFAULT 0,
    guard_count         BIGINT NOT NULL DEFAULT 0,
    enter_count         BIGINT NOT NULL DEFAULT 0,
    total_gift_price    DOUBLE PRECISION NOT NULL DEFAULT 0,
    timeline_json       JSONB,
    keywords_json       JSONB,
    user_stats_json     JSONB,
    gift_stats_json     JSONB,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- =========================
-- song_requests
-- =========================
CREATE TABLE IF NOT EXISTS song_requests (
    id              BIGSERIAL PRIMARY KEY,

    room_id         BIGINT,

    singer          TEXT,

CREATE INDEX IF NOT EXISTS idx_song_requests_session ON song_requests(session_id);

-- =========================

CREATE TABLE IF NOT EXISTS changelog_entries (
    id              BIGSERIAL PRIMARY KEY,

);

CREATE TABLE IF NOT EXISTS alert_logs (

```

### 17.1 `danmaku_events` 写入策略定案

逐条 `INSERT` **会成为瓶颈**。

必须采用：

- recorder 内存 buffer
- 批量 flush
- 单批事务写入
- 优先使用 `COPY` / Binary Import

建议参数：

- batch size：`500 ~ 2000`
- flush interval：`1 ~ 2 秒`
- 强制 flush 时机：
  - batch 满
  - session 结束
  - 进程退出
  - 账号切换 / 录制器停止
